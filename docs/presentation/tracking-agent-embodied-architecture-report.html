Embodied Agent Architecture

Tracking Agent 技术架构报告

一个运行在 robot / Pi 侧的 chat-first embodied agent kernel。 项目当前的重点不是继续堆 tracking 特例，而是把持续感知、单轮决策、状态单源和 skill 插拔化收敛为清晰、可演进的系统骨架。

Inspection Scope

本报告基于仓库实态检查整理，聚焦系统设计、边界划分、组件职责、状态与数据流、部署方式以及扩展路径，不下钻到函数级实现。

项目目标与问题 framing

目标不是单点 tracking demo

仓库正在从 tracking-specific runtime 收敛为通用 embodied agent kernel，tracking 和 speech 是当前最明确的能力样板。

要解决的是系统问题

持续感知、会话状态、多轮对话、技能编排、结果展示需要稳定协作，但不能演化成一个过度工程化的 runtime 框架。

关键约束

perception 只负责写世界状态；runner 只负责单轮处理；viewer 只负责可视化；skill 以 capability module 的方式挂接。

架构原则

Chat-first, not perception-first

turn 由聊天、脚本或 loop 事件触发，perception 只提供 grounded context。

Perception is the only always-on subsystem

只有 perception 常驻运行并维护 observation window。

Single runner path

所有 turn 最终归并到 PiAgentRunner 的同一条主链。

Single persisted state truth

session.json 是 agent-owned state 的主真相源。

Skills are ordinary modules

新增能力应扩展 skill，而不是改造 backend 主干。

系统全景

这个项目最适合被理解为三条运行平面：持续感知平面、事件驱动的 turn orchestration 平面，以及只读展示平面。

Continuous Perception Plane Always-on

scripts/run_tracking_perception.py -> backend/perception/service.py -> persisted snapshot / keyframes

Turn Orchestration Plane Event-driven

backend/cli.py -> PiAgentRunner -> route context / skill context -> Pi or deterministic skill entry

Presentation Plane Read-only

agent_viewer_stream 聚合 session + observation + viewer modules，经 websocket 提供给 React viewer。

高层数据流

Camera / Video

-> Perception Process

-> perception snapshot.json + keyframes

 

User / Script / Tracking Loop

-> backend/cli.py

-> PiAgentRunner

-> Pi or direct skill path

-> session.json + skill_cache

 

session.json + snapshot.json

-> viewer stream websocket

-> tracking-viewer app

总体架构分层

Layer 代表模块 职责 为什么存在
Trigger Layer robot-agent chat、tracking loop、脚本入口 产生 turn 把“何时需要 agent 响应”从内核里剥离出来
Turn Orchestration Layer backend/cli.py、backend/agent/runner.py、pi_protocol.py 统一处理一次事件 避免多条并行决策主链
State Layer active_session.json、session.json、perception snapshot 落盘与共享状态 保证单一真相源与可调试性
Capability Layer skills/tracking、skills/speech、skills/web_search 实现具体能力 让 backend 保持通用
Presentation Layer agent_viewer_stream.py、apps/tracking-viewer 聚合并展示状态 提供演示与观测，而不侵入执行链
Continuous Perception Layer run_tracking_perception.py、LocalPerceptionService 持续写 observation 为 turn 提供 grounded world context
核心不是“模块很多”，而是每个模块的责任边界足够窄且互相可解释。

主要组件与职责

组件 代表路径 主要职责 架构意义
Perception Service backend/perception/ 维护 observation window、保存 keyframe、生成 persisted snapshot、提供 CLI 读取 只做感知，不做高层决策
Agent Runner backend/agent/runner.py 接收 turn、生成上下文、调用 Pi 或 direct path、应用返回 payload 唯一主处理链
Pi Protocol backend/agent/pi_protocol.py 定义 reasoning plane 的输入输出边界 把 LLM 决策面与本地状态更新解耦
Session Persistence backend/persistence/ 维护 session.json 与 active session 单一状态真相源
Skill Surface backend/skills.py discovery、route summary、turn context、direct path、viewer module 聚合 让 capability 扩展不污染内核主干
Tracking Skill skills/tracking/ target init、continue tracking、memory rewrite、viewer module 当前最成熟的 embodied capability
Speech / Web Search skills/speech/、skills/web_search/ TTS 与外部信息能力样例 证明 skill 机制不是 tracking 专用
Viewer Stream + App backend/agent_viewer_stream.py、apps/tracking-viewer/ 将 session、observation、module payload 聚合后可视化 形成 read-only projection layer

Skill 接口为什么关键

build_route_summary

让每个 skill 提供“我现在大概处于什么状态”的摘要，帮助 turn routing。

build_turn_context

让每个 skill 只暴露自己真正需要的 specialized context，而不是让 Pi 阅读整份 session 文件。

process_direct_init / process_direct_turn

为脆弱、确定性强的路径保留 LLM 外的稳态入口。

schedule_rewrite / build_viewer_module

把慢操作和展示扩展都下沉给 skill 自己。

Tracking Skill 的位置

它是单轮 skill，不拥有 perception loop

skills/tracking/SKILL.md 明确说明 tracking 是 single-turn skill，perception loop 和 tracking runtime loop 不属于 skill 本体。

deterministic 入口比自由推理更重要

init / track 走固定入口脚本，降低 fragile workflow 被 LLM 发散的概率。

memory rewrite 脱离关键路径

确认目标后先返回 turn 结果，再异步启动 rewrite worker，优先保证交互响应性。

Tracking 是样板能力，不再是定义整个 backend 形状的“框架中心”。

Embodied Agent Loop

1

Perception 持续写世界状态

camera / video 输入被采样并形成 observation、detections、keyframes 和 persisted snapshot。

2

外部事件触发一次 turn

来源可以是用户 chat、tracking continuation loop、或脚本入口。

3

Runner 聚合上下文

读取 session.json 与最新 perception snapshot，生成 route context 和 skill context。

4

Planning / Routing

deterministic path 直接进入 skill；否则由 Pi 在当前 enabled skills 中完成单轮路由与推理。

5

Action / Result

skill 返回统一 JSON payload，包含 session_result、skill_state_patch、robot_response 等结构化结果。

6

Persistence + Projection

runner 应用状态更新；viewer stream 从持久化状态生成可视化投影；必要时异步进行 rewrite。

Perception / Planning / Action 的实际含义

Perception

以 LocalPerceptionService 为中心，关注 frame、detections、window、snapshot、keyframe 保存。

Planning

以 PiAgentRunner + Pi routing 为中心，决定该轮应该走哪个 skill、采用 direct path 还是 Pi path。

Action

当前更偏 capability result，例如 tracking 的 track / wait / ask，以及 speech 的 TTS 输出，而不是完整机器人控制栈。

为什么这不是“一个大 while 循环”

仓库把持续感知和事件驱动 turn 明确拆开。这样做的结果是，系统既保留了 embodied 状态感知能力，又避免把所有职责糅成一个难以替换、难以验证的大 runtime。

Perception 常驻 Turn 单次处理 Viewer 只读 Rewrite 异步 Loop 只是触发器
这个 loop 的关键不是“自动化很多”，而是每次 turn 的上下文与状态边界都足够明确。

Runtime / Backend / App / Skill / Data Flow 关系

视角 代表对象 关系描述 架构含义
Runtime perception、tracking loop、viewer stream、frontend 一组共享同一 state-root / session_id 的长期进程 职责清晰，可独立重启与组合
Backend backend/ perception、runner、persistence、viewer aggregation 所在的内核层 把系统主线维持在一个最小可解释闭环里
Skills skills/<name>/ 实现具体能力与专属 helper，不改变 backend 的核心形状 支持“继续长能力，而不是继续长框架”
App apps/tracking-viewer 通过 websocket 消费聚合状态，不直接控制业务主链 UI 与内核解耦
Data Flow session.json、perception snapshot、artifacts 共享文件状态连接多个进程与视图 减少依赖，增强可调试性
Active Session

active_session.json 指向当前活跃会话，是 perception、viewer、CLI 的共同入口索引。

Session Truth

sessions/<id>/session.json 保存 latest_result、history、recent_frames、skill_cache 等 agent-owned state。

Perception Snapshot

perception/sessions/<id>/snapshot.json 保存最新 observation、window、stream status 与 saved keyframes。

Turn Artifacts

.runtime/pi-agent/requests/... 保存 route context、skill context、prompt 等调试友好的 turn 证据。

Viewer 为什么重要但不是主角

backend/agent_viewer_stream.py 做的是状态聚合，而不是业务驱动。它把 session、latest frame、stream status、以及每个 skill 的 viewer module 合并为统一 payload，再交给 React app 展示目标框、memory 和 conversation history。

这意味着前端是一个 read-only projection layer：对展示非常重要，但不会反向塑造内核执行方式。

当前系统通过本地共享文件和轮询完成协作，简单直接，但并未追求分布式吞吐或强一致消息语义。

当前推荐部署视图

Process 1

Perception

持续读取 camera / video，运行 detection / tracking，向共享状态写 observation。

Process 2

Tracking Loop

轮询 session 与 perception，在目标已绑定时触发 continuation turn。

Process 3

Viewer Stream

把聚合后的 agent/perception/module 状态通过 websocket 推给前端。

Optional

Frontend

React + Vite 演示界面，负责目标框、状态、记忆和会话记录的可视化。

为什么这种部署合理

职责分离

perception、loop、viewer 可分别重启和定位问题，不会形成单点“超级进程”。

Viewer 可选

前端不再是主流程依赖，适合 headless 运行和演示运行两种模式。

状态共享简单

所有进程共享同一个 state-root 和 session 路径，降低部署门槛。

适合 robot / Pi 侧

不需要额外消息基础设施即可运行，贴合当前仓库的轻量化目标。

扩展性判断

已经具备的能力

新增 skill 可以放在 skills/<name>/，由 backend 自动发现，并在 route context、viewer module、direct path 等协作点自然接入。

当前刻意不做的事

不引入复杂 plugin lifecycle、不上消息总线、不把 loop 升级为总控中心，以避免再次回到过度工程化路径。

最重要的结论

仓库已经具备“继续长能力而不是继续长框架”的结构条件。

Tradeoff 01

Chat-first vs perception-first

当前偏 agent 对话与事件驱动，适合 grounded interaction；如果未来走高频闭环控制，还需单独设计更硬实时的 control plane。

Tradeoff 02

Single session truth vs 多份 memory

调试与可解释性更强，但当前更偏单机和单活跃 session 形态，不是天然的分布式共享状态模型。

Tradeoff 03

Direct path vs full LLM mediation

tracking 的 fragile workflow 更稳，但 skill 需要自己维护更明确的 deterministic 入口与 payload contract。

Tradeoff 04

Async rewrite vs inline completion

主 turn 响应更快，但 memory 更新成为 eventual consistency，而不是同步闭环。

部署与扩展策略都服务于一个共同目标：保持内核小而硬，把复杂度推回到 skill 或可选外层进程。

风险、边界与当前成熟度

最成熟的 embodied capability 仍然是 tracking

speech 和 web_search 已经验证 skill plugability，但 embodied 主场景仍围绕 tracking。

状态共享主要是本地文件级协作

这非常适合单机部署和调试，但不等于已经具备多机分布式架构。

Action plane 仍偏结果型输出

当前重点是 grounded 决策与能力执行，不是完整的机器人运动控制系统。

Perception 聚焦 tracking 场景

它已经是独立感知层，但目前仍是视觉 tracking 导向，而非通用多传感器融合平台。

Future Work

01

更强的部署韧性

为 perception、loop、viewer 增加更明确的健康检查、超时与恢复策略。

02

更通用的 capability surface

在保持 skill surface 克制的前提下，接入更多 embodied 能力，而不让 backend 再次业务化。

03

更强的 observability

增加 turn latency、skill routing、rewrite worker、session lifecycle 等指标与运维视角。

04

多设备 / 远程部署能力

如果跨机部署成为刚需，再从本地文件共享迁移到更明确的 service / bus 架构。

汇报结论

tracking_agent 当前最值得强调的成果，不是某一个 tracking 算法细节，而是它已经把 embodied agent 方案压缩成一个可解释、可部署、可扩展的技术骨架：

持续感知只做感知 单轮 runner 只做编排 状态只有一个真相源 skills 通过统一 surface 接入 viewer 只是展示层

这让项目从“为 tracking 服务的一组脚本”演进为“以 tracking 为样板能力的 embodied agent kernel”。对外汇报时，应把这个系统级收敛能力作为主叙事，而不是把注意力放在低层实现细节上。
