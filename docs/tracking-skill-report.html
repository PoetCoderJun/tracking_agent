00
01 / 为什么是 Tracking

Tracking 是 embodied cognition 的试金石

在数字世界里，agent 验证结果的方式很简单：刷新页面，看 API 返回值，检查数据库状态。但在物理世界里，验证要困难得多——因为世界在持续流动，目标会移动、会遮挡、会消失。

Tracking 逼迫 System 2 解决真正的问题

用户说"跟踪穿黑衣服的人"，这句话对聊天机器人来说只需要一个礼貌的确认回复；但对 embodied agent 来说，它意味着一连串必须在物理世界里被回答的问题：画面里有没有这样的人？如果有，是哪一个？如何在整个视频流里持续确认是同一个人？如果他暂时被柱子挡住了，System 2 应该等待、搜索还是放弃？

这些问题没有一个能被硬编码的状态机完美覆盖，因为真实场景的变化是无限的。这正是为什么 tracking 如此适合作为 System 2 的能力样例：它把大模型的拆解、推理、调用、验证能力全部逼到了台前。

数字 Agent 的验证

环境状态可随时重新拉取。结果是离散的、确定的、可回退的。验证成本极低。

Tracking 的验证

目标在连续运动，每一帧都是新的。验证必须依赖感知的持续记录和 System 2 对因果链的追踪。错误不可撤销。

"如果一个 embodied agent 做不好 tracking，那说明它从未真正在物理世界里'理解'过用户的话。"
01
02 / 核心张力

指称接地：从语言到物理对象的绑定

聊天机器人和 embodied agent 的根本分野，不在于有没有摄像头，而在于语言是否能落地到物理世界中的具体对象。用户说"穿黑衣服的人"，System 2 必须把这个语言描述绑定到画面中的某个具体个体上——这就是指称接地（referential grounding）。

接地不是识别，而是绑定

画面里可能同时有三个人穿着深色衣服。System 2 要做的不是"识别出所有穿黑衣服的人"，而是根据用户意图和上下文，选择其中一个作为跟踪目标并绑定它的身份。这个选择可能是基于位置的（"最靠近门的那个人"）、基于时序的（"最开始出现的那个人"），或者基于用户指明的（"ID 为 3 的那个人"）。

一旦绑定完成，System 2 就必须在后续所有帧中维持这个绑定关系。即使目标转身、被遮挡、光线变化，System 2 都要能够基于共享状态和新的感知快照，判断"我跟踪的还是不是同一个人"。

接地的三个层次
  • 语义匹配：画面里有没有符合描述的候选对象？
  • 身份绑定：在众多候选中，哪一个才是用户指的那个？
  • 时序维持：在连续的视频流中，如何确认身份没有交换？
为什么接地必须发生在 System 2 里

纯视觉模型可以给出"画面里有三个人"这样的检测结果，但它无法回答"用户说的'那个人'到底是指谁"。这个问题必须结合语言理解、对话上下文、空间推理和用户偏好来综合判断。这正是 System 2 的慢思考所擅长，而硬编码框架难以穷举的。

03 / 任务拆解

System 2 如何把"跟踪"拆成可验证的步骤

"跟踪"听起来是一个动作，但在 embodied 场景中，它实际上是由多个语义阶段构成的连续任务。System 2 必须能够自主地把用户的高层指令拆解成这些阶段，并在每一阶段完成后验证进度。

STEP 01

理解意图

解析用户描述，确定目标特征、空间约束或候选 ID。

STEP 02

搜索候选

观察当前感知快照，在画面中寻找匹配描述的候选对象。

STEP 03

确认绑定

在多个候选中选定唯一目标，建立跟踪身份并写入状态。

STEP 04

持续跟踪

在多轮对话或事件驱动中持续观察目标位置和运动。

STEP 05

验证闭环

每轮结束后确认目标是否仍在、是否跟丢、是否需要修正。

硬编码框架的问题

传统方式会把上述五步写成一个固定状态机。但当用户说"切换到另一个人"或"先找到他再跟着"时，状态机就会出现未定义的跳转。System 2 的优势在于，大模型可以根据语言理解灵活地跳过、重复或重排这些步骤。

System 2 的优势

大模型自主判断当前处于哪个阶段、下一步该做什么。它可以在 init 之后直接 track，也可以在用户提问时切换到 reply，还可以在不确定时选择 wait。拆解的灵活性来自语言理解，而不是框架预设的跳转表。

04 / Init 回合

从自然语言到物理对象的首次绑定

init 回合是 tracking 中最关键的一步。它不是"开始运行某个跟踪程序"，而是System 2 在物理世界里完成第一次指称接地：把用户说的"穿黑衣服的人"，绑定到画面中的某个具体 track_id 上。

Init 的本质是决策，不是启动

很多系统把"init"理解为一个简单的启动信号：用户说开始跟踪，系统就打开跟踪器。但在我们的设计里，init 是一个认知动作。System 2 必须读取当前感知快照，结合用户的语言描述，在众多检测候选中做出选择，然后把选定的目标身份写入共享状态。

这个决策不能交给一个硬编码的匹配规则来完成，因为用户的描述可能是模糊的（"那个高的人"）、时序依赖的（"刚刚走进来的那个人"）、或需要对话上下文来消歧的。这些都需要 System 2 的慢思考。

Init 回合的输出
  • 选定的目标 track_id
  • 目标的语言描述（供后续验证使用）
  • 确认帧和截图（作为绑定的视觉证据）
  • 写入共享状态的跟踪记忆
"init 不是打开开关，而是在物理世界里写下'我决定跟踪谁'。"
为什么 init 不能交给纯视觉模型独立完成

视觉模型可以检测画面中的所有人，但它无法判断"用户想要的是哪一个"。这个判断必须依赖语言理解、对话历史和用户偏好——这正是大模型在 System 2 里承担的职责。所以 init 回合的设计是：System 2 做选择决策，确定性脚本执行绑定和状态写入，两者分工明确。

04
05 / Track 回合

持续跟踪是不断验证的循环

当目标已经绑定后，用户可能会说"继续跟踪"。这不是一个新的决策，而是要求 System 2 进入验证循环：确认之前绑定的目标是否还在画面中、位置是否合理、身份是否没有交换。

Track 回合的核心问题是验证

在 track 回合中，System 2 读取最新的感知快照和跟踪状态，判断目标是否仍然可识别。如果目标清晰可见，它更新位置信息并继续跟踪；如果目标被遮挡或短暂离开画面，它可能需要基于历史轨迹做预测；如果目标完全丢失或身份发生混淆，它则需要进入恢复或等待状态。

每一轮 track 都不是盲目的重复，而是带验证条件的观察。System 2 必须在每一轮结束时给出一个明确的判断：found / not found / uncertain。这个判断将成为下一轮决策的基础，也可能触发语音反馈给用户。

Track 回合的判断空间
  • 继续跟踪：目标清晰可见，更新位置，保持跟踪。
  • 等待恢复：目标暂时丢失，但可能重现，保持记忆等待。
  • 请求澄清：画面中出现多个相似目标，System 2 不确定跟谁。
"每一轮 track，System 2 都在回答同一个问题：我绑定的那个对象，现在还在不在？"
持续跟踪考验的是状态一致性

track 回合的有效性完全依赖于共享状态里的绑定信息。如果 init 回合没有正确写入目标 ID、确认帧和描述，track 回合就会变成无头苍蝇。反过来，这也说明了为什么状态外化如此重要：System 2 的工作记忆（上下文窗口）会刷新，但外部记忆里的绑定关系是持续的、可靠的。

06 / 异常处理

当目标丢失时，System 2 该做什么

在真实环境中，跟踪失败是常态而非例外。目标可能被障碍物遮挡，可能走出视野，也可能因为光线变化导致检测器漏检。System 2 不能简单地报错退出，它必须像人类一样做出合理的推断和应对。

Wait 不是放弃，而是有根据的暂停

当 System 2 发现目标暂时不在画面中时，它面临几种选择：立刻宣布跟丢、盲目地搜索画面、或者基于历史信息判断"目标可能很快会重新出现"。在我们的设计中，wait 回合代表第三种选择：System 2 选择保持当前跟踪记忆，等待感知层在后续帧中重新发现目标。

这个决策需要推理：目标之前是向画面边缘移动的吗？它消失的位置附近有没有障碍物？当前的场景是否支持目标很快重现？这些判断无法被硬编码规则穷尽，但可以通过大模型对共享状态和感知快照的综合阅读来完成。

System 2 的三种异常应对
  • Wait：基于历史轨迹判断目标可能重现，保持记忆等待。
  • Ask：画面中出现多个候选，向用户请求澄清以重新绑定。
  • Fail：目标长时间丢失且无可信恢复路径，向用户报告失败。
异常处理体现的是信念维持能力

硬编码系统在遇到未定义状态时通常会崩溃或陷入循环。而 System 2 的优势在于，它可以在不确定的环境中维持信念、评估置信度、并在信息不足时选择最小伤害的应对策略。wait 不是无能，而是对物理世界不确定性的合理尊重。

07 / 状态设计

Tracking 为什么需要外部记忆

Tracking 是一个跨越多个 turn 的连续任务。用户可能在第 1 轮说"跟踪那个人"，在第 5 轮说"继续跟踪"，在第 12 轮问"他还在吗？"如果每一轮都从零开始重新理解任务，System 2 就无法建立起任何连续性。

跟踪状态是 System 2 的笔记本

在 tracking 的共享状态中，有几个关键信息必须被显式保存：当前绑定的目标 ID、用于确认身份的参考帧和截图、用户最初的语言描述、以及最新的跟踪记忆。这些信息构成了 System 2 对"我正在跟踪谁"这一信念的外部支撑。

当 System 2 进程重启、超时重试，或者被新的对话事件触发时，它不需要重新猜测用户的意图。它只需要从共享状态中读取这本笔记，就能立即恢复对当前任务的理解。这让它能够像人类一样，在被打断后继续未完成的工作。

Tracking 状态的关键切片
  • latest_target_id：当前绑定目标的唯一标识。
  • target_description：用户最初的语言描述，供后续验证使用。
  • latest_confirmed_frame / crop：绑定目标的视觉证据。
  • latest_memory：跟踪历史的文本摘要，供大模型快速恢复上下文。
"没有外部记忆的 tracking，就像让人在每次眨眼后都重新辨认自己的朋友。"

08 / 与内核的关系

Tracking 是一个 tool，不是框架的一部分

这是理解整个设计最关键的一点：tracking 的实现并没有被编织进 runner 或内核的框架逻辑里。它是一个独立的 skill，通过标准 CLI 暴露给 System 2 调用。runner 不知道 tracking 内部是如何工作的，它只知道 System 2 在某一轮决定调用 tracking tool，并且这个 tool 返回了结构化的结果。

Tool 与框架的解耦

在传统的 embodied agent 系统中，tracking 常常被当作框架的内置能力：框架负责启动感知、启动跟踪器、维护跟踪状态、并在适当时机把结果塞给大模型。这样做的结果是框架越来越重，tracking 的边界也越来越模糊。

在我们的设计里，这个过程被颠倒了：System 2 决定何时调用 tracking tool，tracking tool 只负责执行本轮的 init 或 track 逻辑并返回结果。状态的写入、下一轮的触发、异常的处理，都由 System 2 在 runner 提供的空间里自主完成。

Tracking 的 tool 契约
  • 只处理单轮请求（init 或 track），不维护长期运行循环。
  • 读取 runner 提供的共享状态和感知上下文。
  • 返回结构化 JSON，包含目标信息、决策和状态补丁。
  • 不直接向机器人发指令，所有动作建议由 runner 把关。
"tracking 不是框架在为大模型服务，而是大模型在把 tracking 当作一个可用的身体能力。"
这种解耦的意义

它意味着未来替换 tracking 的实现（比如换一个检测器、换一个跟踪算法），不需要改动 runner 或 System 2 的内核逻辑。只要新的实现遵守同样的 CLI 接口和返回契约，System 2 就可以无缝地继续调用它。这是能力可替换、框架不膨胀的关键。

Closing Note

Tracking 的价值，
在于它暴露了 embodied System 2 的全部张力。

从语言的 grounding 到物理对象的绑定，从单次决策到跨轮验证，从正常运行到异常恢复——tracking 把一个抽象的"慢思考"理念，变成了一个个必须在现实里被回答的具体问题。

当我们说 System 2 能够拆解、思考、调用、验证时，
tracking 就是这一切的第一块试金石。
