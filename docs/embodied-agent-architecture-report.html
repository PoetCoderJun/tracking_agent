Architecture Report

机器人的 Agent

关键不是做一个更复杂的调度框架，
而是反过来思考：什么样的系统边界，
才能让大模型在物理世界里真正承担起慢思考的角色？

核心信条 让大模型在 Agent 里充分释放慢思考能力
当前形态 一个以 Tracking 和 Speech 为能力样例的 Embodied Agent Kernel
设计入口 对话触发，观察环境，调用工具，验证闭环
00
01 / 出发点

我们要做机器人的 Agent

机器人的底层运动控制、反射性避障、电机驱动，是快速的 System 1。而我们的目标是构建它的Agent：一个能够慢思考、拆解任务、调用工具、验证结果的认知决策层。

Agent 应该由大模型本身来驱动

过往做 embodied agent 时，我们习惯于在一个统一的调度框架里写死业务编排逻辑：跟踪目标要写一套状态机，导航要写一套流程控制。结果是，框架越来越重，大模型被压缩成了只负责"最后一步分类"的薄层，它的拆解、推理、验证能力都被硬编码的逻辑扼杀了。

Tracking Agent 的核心理念是反过来的：我们要做的不是"一个会调用大模型的框架"，而是给大模型提供一个安全、确定、可观察的具身环境，让它自己去做拆解、思考、闭环和验证。runner 不是编排器，而是 Agent 的运行容器。

Embodied Agent 的四大能力
  • 拆解任务：把开放式具身目标分解为观察、澄清、计划、执行、恢复等可落地步骤。
  • 思考推理：基于环境输入、System 1 信号、设备状态与历史上下文判断当前世界状态、风险和下一步策略。
  • 调用能力：通过 CLI/API 调用沟通、移动、操作、导航、媒体与其他具身能力来影响世界。
  • 验证闭环：比较动作后的世界变化与预期是否一致，在偏差出现时修正、重试、回退或请求接管。
与纯云端 agent（如 Operator / Computer Use）的根本区别

Operator：数字世界的 Agent

环境是标准化的网页和 API。操作可撤销、状态可重拉、并行探索没有物理风险。它的闭环验证很简单：刷新页面，看结果对不对。

Tracking Agent：物理世界的 Agent

环境是连续、不可逆的物理现实。Agent 必须处理时间连续性和动作后果的不可撤销性。大模型不仅要思考"做什么"，还要在每一轮验证"做了之后世界变成什么样"，并基于这个变化继续推理。

02 / 架构骨架

Agent 的工作空间

系统的全部复杂性，被收敛成给大模型提供的一个安全、确定、可观察的工作空间。没有中央调度器去限制模型的推理，只有清晰的边界来保护物理世界的安全。

Chat

对话触发 Agent 回合

Script

脚本和自动流程触发执行

Interface

外部界面或设备事件进入系统

Continuous Input Layer

Perception

主常驻输入层。持续提供视觉、语音、设备状态与时间连续的世界快照。

Reactive Input

承接 System 1 推理结果或中间数据产物，并把这些数据同样按需保存下来，作为后续推理的输入

Agent Runner

Context Composer

把输入层、历史上下文和任务目标整理成一轮可工作的认知输入。

Runner

大模型在这里拆解任务、推理判断，并通过唯一动作提交权决定是否调用能力。

Memory

保存任务进度、上下文、验证结果和环境认知等，支撑跨轮慢思考与恢复。

Capability & World

Tools / Capabilities

Tracking、Search、Navigation、Actions 等能力通过统一 contract 被调用。

Physical Action

动作真正作用于物理世界；其后果必须重新进入输入层，而不是停留在工具返回值里。

Verification Loop

动作后观察、偏差修正、重试、回退与接管都依赖同一条感知回路完成闭环。

共享记忆脊柱

连接 Agent 与物理现实的桥梁。大模型通过它读取世界、记录思考、追踪任务进度；其他模块通过它共享对同一现实的认知。

Chat-First

对话不是触发一个函数，而是触发一整轮"理解任务—观察现实—调用能力—验证结果"的认知循环。

One Action Authority

后台感知、预警、恢复和评测可以常驻或并行，但真正提交 physical action 的路径只能有一条，避免并发决策污染物理因果链。

One Agent Memory Source

Agent 的长期记忆必须外化。观察流、动作日志可以各自持久，但 agent-owned 的长期状态应保持单一来源，避免平行记忆镜像。

02
03 / 感知层

Agent 的思考质量，取决于观察的充分性

大模型在 Agent 里做的所有判断——"世界现在是什么状态""动作是否生效""下一步该做什么"——都建立在它所能观察到的环境信息之上。如果观察是残缺或过时的，再强的推理能力也会建立在沙子上。

充分观察是 Agent 的前提

想象一下：Agent 刚刚调用了一个让机器人转头的工具，它需要验证这个动作是否成功，以及环境是否按预期发生变化。如果感知进程在 Agent 不活跃时就停止记录，那么当它准备验证时，拿到的可能是动作执行前的旧快照，或者根本没有相关帧。

这就是为什么感知必须是主常驻世界输入层。它不随某一轮推理的结束而停止，而是持续地、忠实地记录物理世界的变化。只有这样，Agent 在任何时候启动思考时，都能基于一份足够新鲜、足够连续的世界快照来拆解任务、推理判断和验证闭环。

但正式的 embodied harness 并不意味着"只有原始感知能常驻"。System 1 的推理结果——例如目标连续性判断、短时异常告警、设备状态判断、低延迟风险信号——也可以进入输入层。关键不是禁止派生推理进入，而是要求这些结果带着时间语义、来源标记和可覆盖性进入同一输入层，作为对世界可读性的补充，而不是形成第二个动作中枢。

核心张力

物理世界连续运转，而 Agent 的思考是离散触发的。感知层的使命就是弥合这条裂缝，让大模型的慢思考能够随时接入"当前现实"。

"Agent 不是全知的，它的智慧受限于感知的窗口。感知必须是那扇永远不会关闭的窗。"
常驻带来的结构性结果

输入归一

Agent 和 tools 不必各自去连接摄像头或解析传感器；同样，System 1 的派生判断也不应绕过输入层直接支配 Agent。原始观测、设备状态和 System 1 推理结果都应汇入同一输入层，避免大模型基于矛盾信息做决策。

验证就绪

当 Agent 执行了一个动作后，它能立即查询动作之后的世界状态，完成"观察-行动-验证"的闭环。没有感知的常驻，验证就无从谈起。

03
04 / 运行层

串行是具身环境里的基本秩序

Runner 不是一个大模型调用器，也不是一个业务流程大编排器。它是 Agent 的运行与动作提交容器：每一轮，它为大模型准备好完整的上下文（观察、记忆、可用工具），让大模型自主地完成拆解、思考、调用工具、验证结果的完整循环，并对 physical action 的提交负责。

Runner 让 Agent 的判断能对自己的后果负责

在物理世界里，Agent 必须验证它的行动是否产生了预期效果。比如它让机器人向左转，接下来它需要观察画面，确认环境是否按预期变化、机器人是否确实转动了。如果这个验证还没完成，下一轮的指令又发了出去，Agent 就会失去对因果链的追踪。

所以 runner 坚持动作提交串行：一个完整的"思考-行动-验证"循环必须结束后，才能开启下一个动作循环。这不是为了限制大模型的能力，而是因为串行是具身环境里的基本秩序。正式 embodied harness 可以并行存在常驻感知、低延迟预警、后台 recovery 和 eval，但一旦涉及 physical action commit，必须保持单一提交权。只有在一个因果干净的物理空间里，Agent 才能可靠地验证自己的推理，并为自己的判断负责。

为什么只允许动作提交串行

并发在数字世界里是效率，在物理世界里常常意味着因果污染。可以并行的是观察、预警、评测和恢复建议；不能并行的是两个独立决策流同时提交身体动作。串行是对物理世界因果律的尊重。

"Runner 不是业务编排器，而是物理动作的串行提交权。"
一轮循环里的四件事

1. 观察

runner 组装当前感知快照、历史对话和外部记忆。

2. 思考

大模型基于上下文自主拆解任务、推理判断。

3. 行动

模型决定调用哪个 tool（CLI/API），runner 把关并执行。

4. 验证

结果和环境变化被写回状态，供下一轮验证预期。

04
05 / 状态层

Agent 需要外部记忆来支持慢思考

大模型的上下文窗口是它的"工作记忆"，容量有限且随每次调用而刷新。但 Agent 的慢思考往往需要跨越多个轮次：拆解任务、追踪子任务进度、记录验证结果、修正之前的假设。这些都需要一个外部长期记忆。

状态是 Agent 的笔记本

假设 Agent 把一个复杂任务拆解成了三步：观察环境、确认当前状态、执行后续动作。执行到第二步时，它需要记得第一步已经确认了哪些线索、排除了哪些风险。如果这些信息只存在于大模型的上下文里，一旦进程重启或 token 超限，整个任务拆解就断裂了。

外部状态就是 Agent 的笔记本。它把任务进度、验证结果、环境认知都显式地写下来，不仅自己能随时查阅，系统里的其他模块——viewer、recovery、人类操作员——也能读到同一页笔记。这让人和机器能够对物理现实达成共识，也让 Agent 的思考可以跨轮次持续推进。

外部记忆支撑的长程能力
  • 任务拆解追踪：记录复杂任务分解后的子步骤和完成状态。
  • 跨轮验证：保存"之前做了什么""结果如何"，供下一轮修正推理。
  • 环境认知同步：让 recovery 进程、viewer、人类操作员共享同一现实。
  • 故障恢复：Agent 重启后能从笔记中断点续传，而不是从零开始。
状态外化的本质

Pi agent 负责思考，但不垄断系统真相。所有跨轮延续的信息——当前目标、最新检测结果、任务拆解进度、验证记录——都被显式地写回共享状态。这不是为了调试方便，而是为了让 Agent 的慢思考能够在一个稳定、可恢复、可共享的认知基础上持续进行。外部记忆让机器人的 Agent 真正拥有了断点续传的能力。

05
06 / 技能层

下游能力是工具，Agent 通过调用工具来闭环

在这个架构中，skill 的下游实现会被打包成标准的 CLI 或 API。Agent 不需要知道摄像头怎么驱动、语音怎么合成，它只需要知道：当我调用某个 tool 时，物理世界会发生什么变化，以及我如何在调用后验证这个变化。

Tool 是 Agent 影响物理世界的接口

大模型在 Agent 里的推理可能是："用户要求我去确认环境状态并执行一个动作，我需要先调用某个 tool 影响世界，再持续观察结果确认闭环。"这些 tool 对 Agent 来说就像函数调用：传入参数、获得结果、观察副作用。

但 tool 本身不能越俎代庖地控制系统。它只负责在被调用时执行自己的局部能力，并把结果返回给 Agent。runner 作为 Agent 的守门人，确保任何 tool 的调用都建立在完整的全局上下文之上，避免多个 tool 同时盲操同一个机器人。

Tool 的四条契约
  • 以 CLI/API 形式暴露，Agent 通过标准接口调用。
  • 不直接控制硬件调度，只响应单次调用并返回结果，动作提交仍受 runner 把关。
  • 必须说清前置条件、副作用和调用后如何验证，避免 Agent 在物理世界里"盲打"。
  • 不维护独立状态，所有跨轮信息由 Agent 写入共享状态。
"tool 是 Agent 伸进物理世界的手，而 runner 确保这只手不会在没有想清楚时就乱动。"
能力的生长方式

当新的下游能力（如导航、抓取）被其他组实现并封装成 CLI/API 时，Agent 不需要改动自己的内核逻辑就能调用它。只要新 tool 遵守"前置条件清晰、单次调用、结果可验证、不维护独立状态"的契约，它就能自然成为 Agent 思考-行动-验证循环中的一环。系统不是因为 tool 变多而变复杂，而是因为 Agent 逐渐拥有了更多可以支配的身体与世界接口。

07 / 当前形态

我们做出了一个 embodied agent 的认知内核

当前项目真正完成的，并不是某一个具体任务，而是一个能够承载 embodied cognition 的最小内核。

内核已经具备的关键特征

这个内核以对话作为 Agent 的入口，让感知保持常驻，使大模型在任意一轮都能接入当前现实。它坚持单一的动作提交路径，让每一次行动都能被放进清晰的因果链中验证。它把跨轮信息写入共享状态，而不是让长期认知漂浮在 prompt 里。

更重要的是，它把 tracking、speech 乃至未来更多能力都收敛成可调用的 tools，而不是框架内置的逻辑。这意味着，内核不是为某一个 skill 定制出来的脚手架，而是一个可以让不同 embodied capabilities 被统一组织起来的认知底座。

这份内核说明刻意强调的是当前仓库的最小形态：先把主常驻输入层、单一动作提交权、单一 agent state站稳。更正式的 embodied harness 在更严苛的延迟目标下，往往还会长出常驻低延迟辅助层与后台 recovery/eval 面，但这些层应建立在上述硬边界之上，而不是反客为主地重新长成一个重调度框架。

Tracking 的角色

Tracking 是当下最能暴露 Agent 特性的能力样例。它要求语言 grounding、持续观察、动作后验证和异常修正——这些都是 embodied cognition 的核心张力。

Speech 的角色

Speech 说明这个内核不只服务于视觉或空间任务。任何能返回结构化结果的能力，都能自然接入同一个 Agent 工作空间。

"我们现在做成的，不是一个 tracking agent，而是一个以 tracking 和 speech 为能力样例的 embodied agent kernel。"

08 / Harness 设计原则

Embodied Agent 不是把 Claude Code 接上摄像头

普通的 coding agent 工作在文件、终端、网页和 API 组成的数字环境里；Embodied Agent 工作在连续变化、部分可观测、带噪声且不可撤销的物理世界里。Embodied Harness 的增量，不是多接几个 tool，而是让 agent 真正具备读世界、跨时间持续工作、在不确定中恢复、并安全影响现实的能力。

普通 Pi Agent / Claude Code

面对的是相对静态、可重读、可撤销的数字环境。代码可以重新读取，网页可以刷新，失败可以回滚，单轮正确性往往比长期连续性更重要。

Embodied Agent Harness

面对的是持续变化、部分可见、动作有代价的真实世界。它必须保证环境输入持续可读、系统真相显式可追踪、行动后果可验证、异常发生后可恢复。

原则 01

Perception 不是按需调用的 tool，而是系统的主常驻世界输入层。agent 读到的应是可验证、带时间语义的环境快照，而不是临时抓取的零散片段。

原则 02

输入层不只接原始传感器，也接设备状态、System 1 推理结果和低延迟告警；但所有派生结果都必须带来源、时间语义和置信度。

原则 03

状态不能只藏在 prompt 里。任务阶段、目标绑定、关键判断和恢复线索都必须外化为共享真相源，支撑跨轮连续执行。

原则 04

时间、恢复和动作约束是一级设计对象。恢复、等待、重绑、评测可以常驻，但物理动作必须由单一 action authority 串行提交。

架构草案：Embodied Harness 的七个结构件

Perception Backbone

持续接入视觉、语音和设备状态，输出结构化 observation，而不是把原始流直接塞给模型。

Reactive Input Layer

承接 System 1 推理结果或中间数据产物，并把这些数据同样按需保存下来，作为后续推理的输入。

Context Composer

根据当前任务裁切最小必要上下文，把环境、历史和状态整理成一轮可工作的认知输入。

Session State Backbone

保存目标、阶段、判断依据、执行结果和恢复线索，成为跨轮唯一真相源。

Serialized Action Runner

负责每一轮的 framing、推理执行和结果解释，并作为物理动作的唯一提交权，让 agent 自己完成拆解、调用与验证。

Capability Surface

把 tracking、speech、navigation 等能力收敛成统一 contract，使新能力可以按同一方式接入。

Recovery & Eval Layer

负责异常恢复、shadow verification、重绑建议、人工接管和回放评测，保证 agent 对物理后果负责。

09 / 对照关系

当前最小内核，不等于正式 Harness 的全部

这两者不是互相否定的关系，而是分层递进的关系：当前仓库先把最小、严格、 failure-visible 的 embodied kernel 站稳；正式 embodied harness 则是在这些硬边界之上，为更低延迟、更长时连续性和更复杂恢复需求增加常驻能力。

当前仓库：最小 Kernel

目标是先把 chat-first、主常驻输入层、单一动作提交权、单一 agent state source 建立起来，让系统在最少结构下就能形成清晰的"观察-思考-行动-验证"闭环。

它强调的是边界清晰、路径单一、失败可见、易于调试，优先避免 runtime 面和编排面的过早膨胀。

正式系统：Embodied Harness

在不打破上述硬边界的前提下，正式系统往往会再长出 Reactive Input Layer、后台 recovery/eval、常驻低延迟辅助能力，以及更明确的动作前检查、异常回退和人工接管面。

它强调的是在连续物理世界里维持可读性、低延迟和恢复力，而不是回到一个 workflow-heavy 的大框架。

不变的硬边界

无论系统是否升级，主常驻输入层、单一 physical action authority、显式 agent state 和可验证 tool contract 都不应被破坏。

可以增加的层

可以增加的是低延迟派生输入、后台 recovery/eval、shadow verification、rebind 建议和人工接管接口，但它们都应服务于同一动作提交权，而不是另起炉灶。

升级触发条件

只有当延迟目标、恢复复杂度、任务连续性要求或多设备协同压力真实存在时，才值得把最小 kernel 向正式 harness 推进。

一个更准确的理解方式

如果说当前仓库是在回答"怎样用最小结构做出一个真正 chat-first 的 embodied Agent"，那么正式 embodied harness 回答的则是"在不牺牲边界清晰度的前提下，怎样让这个 Agent 长时间、低延迟、可恢复地活在物理世界里"。前者是底座，后者是扩展，而不是两套互斥哲学。

09
10 / 未来与边界

Agent 不做 System 1 的事，也不做硬编码编排

一个好的架构，不仅要说明能走多远，更要说明会在哪里停下来。Tracking Agent 的定位非常清晰：它是机器人的 Agent，而不是 System 1；它是大模型思考的运行容器，而不是替代大模型做思考的硬编码框架。

Agent 不做 System 1 的事

底层的运动控制、反射性避障、电机驱动、实时传感器融合，是快速的 System 1。这些领域有成熟的 OS 和实时控制方案。Agent 的职责层级更高：它要在 System 1 提供的能力之上，回答"在当前这个不确定的物理环境里，下一步该做什么"。

如果我们把实时控制、传感器协议封装、硬件抽象都吞进 Agent，边界就会膨胀，大模型的慢思考会被底层细节淹没，对物理连续性的严格控制也会被侵蚀。

Agent 不做硬编码编排

传统框架试图用硬编码逻辑去编排"先做什么、后做什么"，结果是框架越来越重，大模型被压缩成了分类器。我们的选择是反过来的：让大模型 itself 成为编排者。

Runner 提供的不是业务流程编排逻辑，而是一个安全、确定、可观察的具身控制面。单一路径不是框架在限制模型，而是为了让大模型在一个因果清晰的物理空间里，自主地完成拆解、思考、调用和验证，同时把物理动作的提交权保持在一个严格边界内。

明确不做的事情

不做多 agent 并发提交 physical actions

物理动作不可并发。后台观察、恢复、评测可以并行存在，但 Agent 必须以串行方式指挥身体，确保每一轮都能验证结果。

不做实时控制

毫秒级的避障、平衡、电机反馈是 System 1 的事。Agent 只负责秒级或分钟级的决策和任务规划。

不允许隐式记忆

Agent 的外部记忆必须完全显式。没有"方便起见"的内部缓存，所有跨轮信息落盘共享。

10
Closing Note

Agent 不是框架，
而是大模型与物理世界之间的信任协议。

当感知为 Agent 提供充分的观察，runner 为它守护串行的行动与验证空间，tool 以标准接口供它调用，而共享状态支撑它的慢思考与长期记忆——大模型才能在这个安全、确定、可观察的具身工作空间里，真正释放拆解、思考、闭环与验证的能力。

Tracking Agent 正是这种协议的工程化表达：
简洁、有边界、环境扎根、让大模型 itself 来思考。
